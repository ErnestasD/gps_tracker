# ADR-020: BullMQ for the async job layer

**Date:** 2026-07-08 · **Status:** accepted · **Story:** E04-2 (CLAUDE.md rule 10 gate)

## Context

PROJECT_PLAN §5/§6.1 already names the queue split: **Redis Streams** = the ordered raw
pipeline; **BullMQ** = everything async and unordered (notifications, report generation,
exports, **trip-recompute jobs**, webhook delivery). E04-2 is the first concrete BullMQ
consumer: a late/buffered position batch (§3.6) must trigger an authoritative
`trip-recompute(device, window)` off the hot path (the streaming trip engine is stateful
and drops out-of-order records, so it cannot itself reconcile late data). This is a new
runtime dependency, so it needs the rule-10 paper trail.

## Decision

- **Adopt `bullmq` in apps/worker.** It is the plan-sanctioned choice; it runs on the
  Redis we already operate, supports delayed/deduplicated jobs (job IDs), retries with
  backoff, and a graceful worker lifecycle.
- **Queue `trip-recompute`**, job data `{ deviceId, from, to }` (ISO). Job id
  `recompute:{deviceId}:{bucket}` so overlapping late-batch signals for the same device
  and time bucket **dedupe** into one job instead of piling up.
- **Recompute only touches history** (`fix_time < now − guard`); the streaming engine
  owns the present. This avoids a recompute racing the live persister over the same trip.
- **Reuse the worker's ioredis connection** family for BullMQ (a dedicated duplicate, as
  with consumers) — no second Redis.

## Consequences

- **Redis MUST run `maxmemory-policy noeviction`** (plan §6.1 hard requirement — any
  eviction policy silently corrupts BullMQ queues). Documented in the infra runbook; a
  `maxmemory` alert at 75 % already exists. Streams + BullMQ share the one Redis.
- Redis backup posture is unchanged (plan §6.1): queues hold only in-flight jobs;
  trip-recompute is derived from durable `positions`, so a lost job is regenerated by the
  next late batch or a manual re-enqueue — no BullMQ persistence guarantee is relied upon.
- Downstream stories (E05 notifications/Telegram, E06 reports/exports/webhooks) build on
  this same BullMQ layer; this ADR covers the dependency for all of them.
